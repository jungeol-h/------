# 1차 프로토타입 개발 목표 및 기존 레포 전략

## 1. 개발 핵심 목표 (가설 검증 및 시연)
이번 1차 프로토타입의 절대적인 목표는 상급자/심사위원에게 **"본 플랫폼이 어떤 동선과 가치를 가지고 동작하는지 시각적으로 완벽히 설득하는 것"**이다.

* **복잡도 배제 (Logic-Less):** AI 에이전트를 활용하면 다량의 화면(UI)과 컴포넌트를 빠르게 찍어낼 수 있다. 이 강점을 활용해 5개의 직군(학생, 학부모, 강사, 매니저, 컨설턴트)의 대시보드와 화면 이동 동선을 **전부 다 양적으로 구현**한다. 반면, 복잡한 통계 계산식, DB 스키마 정규화, 토큰 기반 인증 등 눈에 보이지 않는 백엔드 로직은 전면 배제하거나 더미(Mock)로 대체한다.
* **시각적 완성도 집중:** 시스템이 실제로 방대한 데이터를 처리하는 것처럼 보이게 만드는 데 집중한다. 차트 라이브러리(Chart.js, Recharts 등)를 적극 활용해 화려하고 꽉 찬 대시보드를 시연하는 것이 핵심이다.

## 2. 기존 레포(CLAUDE 진단 사이트) 연동 전략
기존에 개발되어 있던 진단 사이트 레포지토리는 핵심 로직이 이미 존재하므로, 이것과 신규 시스템을 통합해야 하는 과제가 있다. 하지만 1차 프로토타입의 제한된 시간 안에서 물리적인 DB 마이그레이션이나 레포 통합을 시도하는 것은 극도로 위험하다.

### **[프로토타입 적용 전략: 프론트엔드 뷰(View) 껍데기만 이식]**
1. **완전 분리 및 UI 흉내내기:** 신규 프로토타입 프로젝트에 기존 진단 로직을 가져오지 않는다. 대신 신규 앱의 <진로 탐구 탭>에 `[진단 시작하기]` 버튼을 둔다.
2. **시연용 하드코딩 플로우:** 이 버튼을 누르면 기존 진단 사이트의 프론트엔드 UI만 흉내낸 더미 설문 화면(1~2문항)이 나오고, 완료 시 곧바로 **가짜 데이터로 꽉 채워진 '진단 결과 리포트' 화면**으로 넘어간다.
3. **향후(2단계) 실제 통합 방향:** 실제 본 개발에 착수할 때는 신규 플랫폼과 기존 진단 서버를 API 형태(Microservice)로 통신하게 하거나, 혹은 아예 신규 레포를 Monorepo(모노레포)로 설정하여 기존 컴포넌트를 안전하게 이주(Migration)시킬 예정이다. 프로토타입 시연 시에는 이 구조적 로드맵을 구두로 설명하는 것으로 충분하다.
